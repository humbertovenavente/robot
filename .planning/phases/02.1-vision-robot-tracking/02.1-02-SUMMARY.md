## Decisions Made
- **Mock strategy deviation:** Tests were initially written using `patch.object(vision_confirm._pyzbar, "decode", ...)` which fails when pyzbar is not installed (because `_pyzbar` is `None`). Fixed by patching `vision_confirm._pyzbar` itself with a full `MagicMock()` object that has `.decode` as an attribute. This is the correct pattern for optional-import module mocking.
- **Python 3.9 compatibility:** Used `Optional[Tuple[int, int]]` from `typing` instead of `tuple[int, int] | None` union syntax (not available until 3.10).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed test patching for optional pyzbar import**
- **Found during:** Task 1 GREEN phase
- **Issue:** Original test design used `patch.object(vision_confirm._pyzbar, "decode", ...)` but `_pyzbar` is `None` when pyzbar is not installed, causing `AttributeError: None does not have the attribute 'decode'`
- **Fix:** Changed all pyzbar-dependent tests to create a full `MagicMock()` as a stand-in module (`_make_mock_pyzbar()` helper) and patch `vision_confirm._pyzbar` with the mock object instead of patching a sub-attribute of the real module
- **Files modified:** tests/test_vision_confirm.py
- **Verification:** All 13 tests pass in environment without pyzbar installed
